zig-sdl GUI — Architecture & Design

A living design record: needs, the choices considered, the choices made, the reasons, the lore, the dynamics — captured as they are discussed, built, tested, and mutated.

Project: a small immediate-mode GUI for Zig + SDL3, with app logic in Lua (LuaJIT) and live hot-reload; designed so the same UI code can later run split across an optional socket "umbilical" (desktop ↔ desktop, VPS, or phone).

Built & verified: Phase 1 — native core Phase 2 — Lua + hot-reload debugging (gdb + LuaPanda) pre-commit gate  ·  Designed-for: Phase 3 — umbilical Phase 4 — Android

How this document is maintained. It is meant to grow with the project. Each design topic is written as Considered → Chosen → Why → Status, where Status is one of tested (built & verified), deferred (decided, not yet built), or mutated (a prior decision that changed). New decisions get a row in the Decision log with a date. When something is corrected or mutated, the old reasoning stays (struck or noted) so the why of the change is legible later. See also the user-facing README and the source under src/.

1. The need

A GUI that is simple to start, but built to scale into something unusual: mobile-first in target (an Android phone), desktop-mainly in workflow (a Debian/Windows laptop with no touchscreen), connected by an "umbilical" so the app can evolve on the fly while it runs.

NeedWhat it forces in the design
Simple, but expansion-readyA tiny core with clean seams; everything advanced is additive, never a rewrite.
Mobile + desktop from one codebaseUnified pointer (mouse and touch); DPI scaling; no platform forks in UI code.
Layout that doesn't fight youA layout engine that resolves rectangles from intent, not hand-placed coordinates.
Evolve on the flyApp logic that reloads instantly — pushing toward a scripted (not compiled) logic layer.
Umbilical dev (desktop→phone/VPS)The UI's input and output must be serializable data so they can cross a socket.

2. The central idea — a GUI is a function whose edges are data

Everything else is a consequence of one decision:

frame(state, input_events, ui) → (state', command_buffer)
                                  -- app.lua calls ui.button(...), ui.slider(...)

Both the input going in and the command buffer (the draw list) coming out are plain data — no renderer pointers, no OS handles. The instant that holds, the function stops caring where it runs or where the pixels land. Three deployments collapse into one code path.

The layout engine and the command buffer reinforce each other: widgets don't paint, they append commands while a layout engine resolves their rectangles — and that command buffer is exactly what the umbilical would serialize. The shape that eases layout is the shape that enables the network. This is the microui insight.

3. Decision — immediate-mode, not retained-mode

ConsideredChosenWhy
Retained-mode (persistent widget tree + callbacks, Qt/DOM-style) Immediate-mode Retained mode keeps two sources of truth (your data + the widget tree) that must be synced — where most GUI bugs live. Immediate mode re-declares the UI from state every frame, so the UI is always a faithful image of state. Less bookkeeping; trivially data-driven; pairs with "UI = function".

No desync, tiny core, and the per-frame redraw cost is nothing for a 2D widget UI.

Each widget call computes its rect, reads the pointer, emits commands, returns a result (if ui.button("+") is the click). tested

Lore. Immediate-mode UI was popularized by Casey Muratori (Handmade Hero) and Omar Cornut's Dear ImGui. The mental shift: think "what should this frame look like," not "respond to this event." The cost is that state is yours and widget order matters for identity (§12).

4. Decision — containers + row sizing specs

ConsideredChosenWhy
Hand-placed x/y coordinates · Flex/constraint solver (two-pass) microui-style rows with per-column width specs Hand-placed coordinates don't survive resize/DPI and don't scale to real UIs. A full flex solver is more expressive but more code and harder to keep immediate-mode-pure. Row specs are tiny, deterministic, single-pass, and resolve perfectly against the viewport.

A column width spec is read as: >1 → fixed logical pixels (×DPI scale), 0..1 → fraction of available width, ≤0 → "fill" (share the leftover space). A row with no specs is one full-width column. Resolution happens once when the row is declared.

See beginRow() / next() in src/ui/core.zig. tested

5. Decision — widgets are userland, not a fixed catalog

The toolkit's value is the public widget-building API on the context, not a closed set of widgets. A "widget" is any function that uses three families of primitives:

Layout

next(), row(specs,h) — ask the engine where you go.

Interaction

getId(), hovered(), buttonBehavior() — query the unified pointer against a rect/id.

Drawing

drawRect/drawBorder/drawText/drawImage, measure() — emit commands.

Composition

Widgets call widgets. A custom one is ~10 lines, built from the same calls — in Zig or Lua.

widgets.zig holds example widgets (label/button/checkbox/slider); nothing in core/ or render/ depends on them, so you replace or extend freely. Built-ins are not privileged.

"Flexibility is a must" — the user wanted definable/composable widgets, not pre-baked ones. This also leaves the door open for a later declarative/data-driven layer on the same primitives. tested (custom widgets demonstrated in Lua).

6. Decision — native core in Zig

ConsideredChosenWhy / trade
C + microui · Rust + egui · Go · Odin Zig (+ SDL3) This project leans on Zig's two genuine superpowers: best-in-class C interop (translate-c consumes SDL/Lua headers directly — no bindings to maintain) and best-in-class cross-compilation (trivial Android/Windows targets — exactly the mobile-first + VPS-build story). No GC suits per-frame immediate mode. Cost: pre-1.0 churn (see §17).

The alternatives, fairly (recorded so the road not taken stays legible):

  • Rust + egui — the strongest competitor: egui already is this paradigm and runs desktop/web/Android. You'd ship faster but adopt a framework instead of owning a minimal one; C interop is more ceremony.
  • C + microui — lowest-risk proven foundation; could even be bound from Zig to get both. Manual memory; harder Android/build setup.
  • Go — great for the headless server half; weak for the SDL render half (cgo cost, GC in the loop).
  • Odin — a true Zig peer for this (great C interop, gamedev focus); smaller community, also pre-1.0.

The architecture itself is language-agnostic — the language choice changes build/interop/maintenance ergonomics, not the design.

7. Decision — app logic in Lua (LuaJIT)

The architecture already drew a seam at "the part that evolves." Lua makes that seam dynamic: the native Zig core (SDL, command buffer, layout, widget primitives) is compiled rarely; the frame() logic is a script.

ConsideredChosenWhy
Native compiled app.zig (restart to evolve) · Native hot-reload via reloadable .so Lua-scripted app.lua Turns the deferred, heavy native hot-reload into something free and instant: reload = re-read a file. The umbilical can later ship text, not cross-compiled binaries. The dynamic boundary sits exactly where the user keeps pointing ("evolve on the fly").

It converts the hardest goal (live evolution) into the cheapest mechanism, and it mutated a prior decision: "no hot-reload in v1 / fast-restart only" became "Lua hot-reload is in v1" once Lua was chosen. mutated → tested

Cost paid up front: a Zig↔LuaJIT binding (script/lua.zig) that reflects the widget API into a Lua ui table. Per-frame Lua is a non-issue for a widget UI (LuaJIT is fast; calls stay coarse).

Similar prior art: game engines pairing a native core with Lua scripts; especially Löve2D (Lua + SDL). This is "Löve2D-but-immediate-mode-GUI-with-an-umbilical."

8. Decision — the umbilical is optional, in two senses

"Optional" is not a feeling; it is structural.

Runtime-optional

The frame loop talks to a Transport interface (sendInput/recvInput/sendCommands/recvCommands). The Local implementation is a zero-cost in-process pass-through (stash a buffer, read it back — no sockets, no serialization). "Not networked" is just the Local impl of the same interface.

Compile-time-optional

Nothing in ui/ or render/ imports net/ — they know only the interface + the data types. A local-only build can exclude net/ behind -Dnet=false and contain zero socket code. You opt into the umbilical; you never opt out.

deferred — the Transport/net/ layer is designed, not yet built (Phase 3). The -Dnet build option already exists in build.zig as the placeholder for this seam.

9. Two umbilical channels

Logic-push is the cheaper, higher-value first step (Phase 3): it is literally hot-reload where the file arrives over a socket. It largely dissolves the "VPS must compile Zig" question — logic is text. Remote-render earns its keep only for "a server driving many thin clients." deferred.

10. VPS roles

Because logic is text, a VPS rarely needs to compile:

  • Lua distribution authority — holds app.lua, pushes it to clients; instant, fan-out, no compile.
  • Cross-compile/build server (native host only) — runs zig build -Dtarget=aarch64-linux-android so the laptop never sets up the NDK; needed only when the native core changes.

Caveat (security): pushing code to run on a shared box is RCE-by-design — fine for a personal umbilical; a public/multi-client server needs auth + a sandboxed Lua env (tracked with deferred TLS/multi-session work). deferred

11. Dynamics — the frame pipeline

  • beginFrame resets the per-frame arena + command buffer, ingests input events into the pointer model (edge flags pressed/released), and sets the layout body from the viewport (DPI-scaled padding).
  • frame() (Lua) declares the UI; widgets append commands. Text strings are copied into the arena so the buffer stays valid until render.
  • endFrame clears the active widget on pointer-release.
  • render walks the command buffer into SDL draw calls; present with vsync.

12. Dynamics — identity & the unified pointer

Interaction IDs come from hashing a widget's label seeded by an id-stack (getId, pushId/popId). hot_id = pointer is over it; active_id = pointer pressed down on it; a click is press-on-it AND release-on-it. This is what makes if ui.button("+") correct across the press→release pair.

Specificity / gotcha. Identity is label-based, so two widgets with the same label in the same scope collide — use pushId(i) in dynamic lists. The demo keeps labels unique (-, +, Reset) so it "just works." This is the classic immediate-mode trade and is documented, not hidden.

Unified pointer (touch-ready). SDL_EVENT_MOUSE_* and SDL_EVENT_FINGER_* both map to the same pointer_down/up/move events (finger coords are normalized 0..1 → multiplied by output size). Widgets never know whether a touch or a mouse drove them. tested

13. Dynamics — text is measured, then cached

Layout needs text width (a button is as wide as its label), but text commands carry the string, not pixels. So:

  • The core measures via a backend-supplied measure callback (MeasureFn) — a single indirection that lets a future headless server measure with TTF (no video subsystem) while a client rasterizes. Layout stays pixel-exact in both.
  • The backend rasterizes glyphs once with TTF_RenderText_Blended and caches the texture (keyed by string), tinting per-draw via SDL_SetTextureColorMod — so the same white glyph texture serves any color, and re-rendering happens only for new strings.

v1 simplification (a deliberate, recorded shortcut): a single font pixel size, re-opened only when DPI scale changes (the cache is then cleared). HIGH_PIXEL_DENSITY is intentionally off so render-output size == window size == mouse-coordinate space (no point/pixel mismatch).

14. Dynamics — hot-reload

The loop checks the script file's modify_time each frame (via SDL's SDL_GetPathInfo); on change it re-reads the file (SDL_LoadFile) and re-runs the chunk, which redefines the global frame. The state table persists across reloads (it's held by a registry ref), so values like count survive an edit. A pcall guard turns a syntax/runtime error into an on-screen message instead of a crash; fixing the file clears it.

Chunks load under an @-prefixed name so tracebacks and the Lua debugger resolve to the real path, not [string "..."]. See loadFile/maybeReload in script/lua.zig. tested — verified by a unit test (edit→reload→new output; bad-edit→error view) and demonstrated live (editing a running window changed its title and added a button, state preserved).

15. Dynamics — debugging (two independent debuggers)

  • Native (gdb) — the Debug build has symbols; halts the whole process at Zig breakpoints.
  • Lua (LuaPanda) — opt-in via ZIGUI_LUA_DEBUG=1; the embedded Lua connects out to a VS Code LuaPanda session on 127.0.0.1:8818 (loopback only by design). Best-effort: no listener → quiet no-op. Started once at startup so it survives hot-reloads. A LuaPanda breakpoint parks inside frame() (the window pauses, resumes on continue) while gdb still sees the process running.
  • Both at once — the compound "Dual debug" config runs gdb + LuaPanda together.

Specificity: a breakpoint must target the statement, not a one-line if … then … end (the if is polled every frame). This is exactly why the project keeps each statement on its own line (stylua collapse_simple_statement = "Never", §16). tested

16. Dynamics — the pre-commit gate

Enabled via git config core.hooksPath .githooks. On commit, .githooks/pre-commit:

  1. resolves zig (PATH or ~/apps/zig — GUI git clients often lack PATH),
  2. autoformats staged .zig (zig fmt) and staged .lua (stylua, excluding the vendored LuaPanda.lua), re-staging what changed,
  3. checks: zig build then zig build test — a failure blocks the commit.

Keeps main always-formatted and always-green; commits go straight to main (no branch) because the gate is the safety net. tested

17. Context — the Zig 0.17.0-dev tax

Targeting a dev build meant absorbing in-flight std/toolchain changes. Recorded so the why behind unusual choices is clear:

Change in 0.17-devWhat we did
@cImport removedC interop via a build translate-c step over src/cdefs.h (SDL3 + ttf + image + LuaJIT in one module).
std.ArrayList is unmanagedBuffers carry the allocator explicitly (cmds.append(gpa, …)); init via .empty.
Build API: createModule/root_module; linking moved onto the modulemod.linkSystemLibrary(...); addExecutable(.{ .root_module }).
std I/O + args mid-rework (Io param)Config read via SDL_getenv; file I/O via SDL (SDL_LoadFile/SDL_GetPathInfo) — sidesteps the churn entirely.
Lua C-API entries are macrosCall the real underlying functions (lua_createtable, lua_pushcclosure, lua_settop, …) rather than the macro spellings translate-c can't emit.
Wisdom. Pin the toolchain; expect periodic fixups on upgrade. The interop/cross-compile wins are worth the churn for this project — but the churn is real and is the main standing cost.

18. Module map & dependency rule

The rule that protects the whole design: ui/core.zig imports only ui/command.zig (plain data) and calls no SDL. Widgets sit on core; the backend and the Lua binding sit above; main wires them. So the entire UI computation can be lifted out and run headless on a server (Phase 3) without touching widget code. Files: command · core · widgets · theme · sdl_backend · lua · main.

20. Roadmap

  1. done Phase 1 — native core: command buffer, layout, widgets, SDL backend, native demo.
  2. done Phase 2 — Lua app logic + file-watch hot-reload + error guard.
  3. next Phase 3 — umbilical: net/ (length-prefixed codec) + Transport{Local,Tcp}; logic-push first (smallest, reuses hot-reload), remote-render later; desktop↔desktop, then VPS, then phone via adb reverse.
  4. later Phase 4 — Android: native host packaged with SDL3 (NDK + gradle shell + APK assets).
  5. later Multi-session server, TLS/auth, scrolling/text-input widgets, declarative widget specs.

All additive on the same frame/command-buffer boundary and public widget API — not rewrites.
